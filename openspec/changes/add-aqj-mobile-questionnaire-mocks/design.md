## Context
AQ日本語版（一般成人用）50問アンケート（`AQ_Adult_Japanese.md`）を、モバイル端末から負担少なく回答できるWebフォームとして試作する。
対象環境は GitHub Pages 上での静的ホスティングであり、サーバサイド処理やデータベースは前提としない。

Six Degrees のモバイル調査デザインガイドは、スマートフォンでの調査において以下を推奨している:  
- 設問テキストおよび全体の設問数をできるだけ短く・シンプルに保つこと  
- 1ページ1問のフローを採用し、長いマトリクス表を避けること  
- 横スクロールが発生しやすい横並びラジオボタンではなく、縦方向ラジオボタンやドロップダウンを用いること  
- 画像・グラフィックを控え、読み込み負荷を下げること  
- 複数のデバイス/OS/ブラウザで表示を検証すること  
参照: `https://www.six-degrees.com/effective-mobile-survey-design-for-market-research/`

Qualtrics および Questant のマトリクス質問ガイドは、PC向けの横長マトリクス表をそのままスマホで表示せず、  
1行ごとに独立した設問として扱うなど、スマホ前提のレイアウトに変形することを推奨している。  
参照: `https://www.qualtrics.com/support/jp/survey-platform/survey-module/editing-questions/question-types-guide/standard-content/matrix-table/`  
参照: `https://help.questant.jp/hc/ja/articles/900005430346-%E3%83%9E%E3%83%88%E3%83%AA%E3%83%83%E3%82%AF%E3%82%B9-%E3%82%B9%E3%83%9E%E3%83%9B%E5%90%91%E3%81%91%E3%83%87%E3%82%B6%E3%82%A4%E3%83%B3`

本変更では、AQ-J の表形式（No / 質問文 / 1〜4）を3種類のモバイル向けUIパターンに変換し、  
UX比較のためのMOCとして提示する。

## Goals / Non-Goals
- Goals:
  - AQ_Adult_Japanese.md の設問文・順序・4件法を保持したまま、モバイルフレンドリーな3種の回答UIを用意する。
  - 3種のMOCを GitHub Pages 等の静的ホスティングで個別URLとして公開できるようにする。
  - 横スクロール不要・縦方向操作中心・タップしやすい UI を実現する。
- Non-Goals:
  - 本変更ではAQスコア計算ロジックや結果フィードバック画面の実装は必須としない（必要であれば別変更で扱う）。
  - 回答データのサーバサイド保存や本番運用に向けたセキュリティ要件は対象外（MOCとしてのUX検証が目的）。

## Decisions
- Decision: 静的HTML/CSS/バニラJavaScriptのみを用いる
  - 理由: GitHub Pages でのホスティングを容易にし、依存を最小限に抑えるため。
- Decision: 共通のCSS（例: `aqj.css`）でタイポグラフィとレイアウトを定義し、各MOC HTMLは同じデザイン言語を共有する。
- Decision: ドロップダウン形式では `<select>` 要素を用い、1ページ内に50問を縦に並べるが、各設問をカード状に区切り、スクロール中の見失いを防ぐ。
- Decision: ドロップダウン形式では 5〜10問単位で折りたたみ/タブ/ページ区切りを設け、長大なスクロールを避けつつセクション見出しに「Q1〜Q10」のような範囲を表示する。
- Decision: 一問一回答形式では、1画面1問＋縦方向ラジオボタン＋進行度インジケータ（残り問数やプログレスバー）を配置する。
- Decision: カルーセル形式では、1問ごとの「スライド」を横方向に並べ、CSSの scroll-snap もしくはシンプルなJavaScriptでのスライド遷移を採用する。
- Decision: カルーセル形式では各スライドに常時「現在の質問番号/全体50問」を明示するヘッダーまたはフッターを配置し、位置の把握を容易にする。
- Decision: いずれの形式でも、選択肢は縦方向に並べるかドロップダウンにまとめ、横方向に隠れた選択肢が存在しないようにする。

## Risks / Trade-offs
- Risk: 50問というボリューム自体が長く、どのUIでも回答負荷が高くなりうる。
  - Mitigation: 一問一回答形式やカルーセル形式では進行状況を明示し、心理的負担を軽減する。MOCの評価時に完了率や主観的負担を比較する。
- Risk: カルーセル形式は実装がやや複雑になり、アクセシビリティ確保が難しくなる可能性がある。
  - Mitigation: キーボード操作とスクリーンリーダー利用時の挙動を設計段階から考慮し、フォーカス管理やARIAラベルを簡潔に実装する。

## Migration Plan
1. 本変更で3種のMOCと仕様を追加し、GitHub Pages 上でのテスト環境を構築する。
2. 実ユーザー/ステークホルダーからのフィードバックを収集し、どのUIパターンを本線とするかを決定する。
3. 将来的に本番用のAQ-J Web実装を行う際には、選定したパターンの仕様を `specs/aqj-web-questionnaire/spec.md` に反映し、不要なMOCをアーカイブする。

## Open Questions
- 回答データをどの程度構造化して取得・エクスポートしたいか（例: JSONダウンロード、クリップボードコピーなど）を本変更の範囲に含めるかどうか。
- AQスコアの即時計算・フィードバック（例: スコアレンジごとの説明文）をMOCの一部として見せるべきか、別変更で扱うか。
